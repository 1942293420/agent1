# 多Agent框架四大特征竞品对比（市场调研 t1）

> 调研范围：AutoGen、CrewAI、LangGraph、MetaGPT、ChatDev 2.0
> 调研维度：并发模型、熔断机制、实时推送、编排模式
> 日期：2026-06-24

---

## 一、AutoGen（Microsoft）

| 维度 | 分析 |
|------|------|
| **并发模型** | 基于 `autogen-core` 事件驱动编程模型，Agent 间通过异步消息（pub/sub）通信。`Team` 支持 `RoundRobin`、`Selector`、`Swarm` 等预设拓扑，`Swarm` 模式允许 Agent 间动态 handoff，本质上为单进程异步并发——多 Agent 在同一事件循环中协作，不提供原生多进程/分布式执行能力。 |
| **熔断机制** | 无原生熔断器。核心层通过 `CancellationToken` 支持手动取消，`max_turns` 限制对话轮次作为兜底，但没有超时自动中断、错误率阈值熔断等生产级保护。需开发者自行在 `MessageHandler` 中包装 try/except 并实现退避逻辑。 |
| **实时推送** | AgentChat 层提供 `StreamingMessage` 协议，支持工具调用和文本生成的逐步推送（token 级流式输出）。团队内部通过 `on_messages_stream()` 生成器向外暴露流式事件，前端可消费该流实现 UI 实时更新。但未内建 WebSocket/SSE 推送通道，需自行桥接。 |
| **编排模式** | 事件驱动 + 设计模式库（`design-patterns`）：`Handoff`（Agent 间任务移交）、`Group Chat`（广播-响应的多 Agent 讨论）、`Magnetic-Orchestrator`（基于向量相似度的动态匹配路由）。编排逻辑由 `Team.run_stream()` 驱动，本质上是一个中心化的流程控制器遍历 Agent 拓扑图。 |

## 二、CrewAI

| 维度 | 分析 |
|------|------|
| **并发模型** | 提供两条路径：`Crew`（高层封装，顺序执行 Tasks 列表——无并发，task 之间通过依赖声明串行连接）和 `Flows`（事件驱动架构，每个 Flow 节点可独立执行，节点间通过 `@listen` 装饰器触发）。Flows 支持多个独立 Crew 并行运行，但框架本身不管理进程/线程池，并发粒度由开发者控制。 |
| **熔断机制** | 无内建熔断器。在 Crew 模式下，`Task` 的 `max_retries` 提供简单重试，`Guardrails` 可做前置输入校验和输出校验，但不是运行时熔断。Flows 模式下节点执行失败会抛异常，需开发者在 `@listen` 回调中自行实现 `tenacity` 或手动退避。 |
| **实时推送** | CrewAI Enterprise（AMP Suite）提供 `Tracing & Observability` 控制面，支持实时日志、指标和调用链追踪。开源版通过 Python 标准日志输出，无 WebSocket/SSE 原生支持。`StepCallback` 和 `TaskCallback` 允许在每步/每任务完成时触发自定义回调，可用于桥接外部推送通道。 |
| **编排模式** | 两种模式互补：`Crew` 模式为静态 DAG——预先定义 Agent→Task→Process 链，适合确定性工作流（如文档生成、数据分析）；`Flows` 模式为事件驱动 DAG——通过装饰器 `@start()` / `@listen(flow_method)` / `@router()` 构建动态路由，适合条件分支和循环编排。`Flow` 支持 `orchestrator` 方法做中心化调度，也可用 `@router` 做分布式决策。 |

## 三、LangGraph（LangChain）

| 维度 | 分析 |
|------|------|
| **并发模型** | 基于 Pregel 的大规模图计算模型，StateGraph 中每个 Node 是纯函数，运行时引擎采用 BSP（Bulk Synchronous Parallel）超步执行：同一超步内所有就绪节点并行计算，超步间通过 Channel 传递状态更新。`Send()` API 支持扇出（fan-out）并行——单个节点可派发多个并行子任务，结果通过 `Command` 聚合回状态图中。生产部署时配合 LangGraph Platform 可实现水平扩展。 |
| **熔断机制** | 通过 `Interrupt` 机制实现类似熔断的效果：在关键节点前设置中断点，满足条件（如置信度低、工具调用失败 N 次）时暂停执行并等待人工介入。`checkpointer` 持久化状态到 Postgres/SQLite 后支持超时恢复——长时间未 resume 可视为静默降级。无传统意义上的自动熔断器（如 Hystrix 模式），但 `Command` + `Interrupt` 组合可实现等价语义。 |
| **实时推送** | LangGraph Platform 原生支持 `stream_mode` 多种模式：`values`（状态快照推送）、`updates`（增量状态更新推送）、`messages`（LLM token 流推送）、`custom`（自定义事件）。通过 LangGraph Server 的 SSE 端点直接推送到 Web 前端，支持 LangSmith Studio 的可视化实时调试。开源版通过 `astream_events()` 暴露全部中间事件。 |
| **编排模式** | 有状态图（StateGraph）编排：用 `add_node`（节点）、`add_edge`（固定边）、`add_conditional_edges`（条件分支）构建 DAG/循环图。支持 `subgraph`（子图嵌套复用）、`Send`（Map-Reduce 动态并行）、`Command`（节点主动控制下一步跳转，含 `goto` / `update` / `resume`）。本质上是一个低层级的有状态工作流引擎，编排灵活度最高但使用者需自行构建 Agent 语义层。 |

## 四、MetaGPT

| 维度 | 分析 |
|------|------|
| **并发模型** | 采用角色流水线（Role Pipeline）模式，Agent 按公司 SOP（如 产品经理→架构师→项目经理→工程师）串行执行，同一阶段的多个 Action（如竞品分析 + 需求分解）按 `_watch` 订阅机制在前一个 Action 完成后依次触发。不提供真正意义上的并行执行——多 Agent 协作为顺序消费消息，类似 Actor 模型的单线程调度。 |
| **熔断机制** | 无熔断设计。流程控制依赖 SOP 定义的阶段边界：每个 Role 的 `_act()` 方法执行单个 Action，失败后依靠 `max_react_loop` 限制重试次数防止死循环。遇到 LLM 调用异常时角色会进入 idle 状态等待下一条消息，不会自动降级或跳过。 |
| **实时推送** | 通过 `Environment` 消息总线实现 Agent 间通信——每个 Role 的 `publish_message()` 将结构化文档（如 PRD、API 设计）推送到共享环境，订阅该类型消息的其他 Role 自动接收。`Message` 携带 `cause_by` 溯源标识形成消息链。但环境消息总线是进程内内存实现，无外部推送能力（WebSocket/SSE 需自行封装）。 |
| **编排模式** | SOP 驱动编排（核心哲学 `Code = SOP(Team)`）：通过 YAML 配置文件定义 Role→Action→SOP 的静态流水线。`Team` 类将多个 Role 组装为串行/分支工作流，`Environment` 作为共享上下文空间。AFlow（ICLR 2025 论文）提出了强化学习自动优化 SOP 编排路径，但当前主分支仍为人工预定义 SOP 模式，缺乏运行时动态路由能力。 |

## 五、ChatDev 2.0（DevAll，OpenBMB）

| 维度 | 分析 |
|------|------|
| **并发模型** | 2.0 版本引入 Web 控制台 + Python SDK 双层架构。后端基于 FastAPI，工作流通过 YAML 配置文件定义 Agent 节点拓扑（支持 DAG），任务提交后按拓扑异步执行。多个工作流实例可并发运行（FastAPI 原生 async + 独立 runtime 实例）。ChatDev 1.0 的链式对话拓扑在 2.0 中升级为 DAG 并行能力。 |
| **熔断机制** | 未暴露独立的熔断器 API。后端采用 FastAPI + asyncio 超时控制，SDK 的 `run_workflow()` 可通过 `timeout` 参数限制总执行时间。工作流内节点失败会导致流程中断（fail-fast），不提供部分降级或自动跳过失败节点的机制。Memory 模块持久化中间状态，支持断点恢复。 |
| **实时推送** | Web 控制台提供可视化工作流执行监控：节点状态实时更新（待执行→运行中→完成/失败），每个节点的中间产物（代码、图表、文本）可通过日志面板实时查看。后端通过 WebSocket 推送执行进度到前端，支持人机协同反馈（Human-in-the-loop 节点等待用户评审后继续）。 |
| **编排模式** | "Puppeteer" 编排架构（NeurIPS 2025）：中央编排器用 RL 训练动态选择 Agent 激活序列和分工策略。2.0 产品化后提供可视化画布拖拽编排 + YAML 配置零代码定义工作流。支持 `sequential`（串行链）、`parallel`（分支并行）、`conditional`（条件路由）三种拓扑，节点之间通过共享 Memory 上下文传递数据。独创的 `Sync` 节点类型用于聚合并行分支结果。 |

## 六、综合对比矩阵

| 特征 | AutoGen | CrewAI | LangGraph | MetaGPT | ChatDev 2.0 |
|------|---------|--------|-----------|---------|-------------|
| **并发模型** | 事件循环异步 | 事件驱动+手动并行 | BSP超步并行 | 串行 Actor | Async DAG 并行 |
| **熔断机制** | ❌ 手动 | ❌ 手动 | ⚠️ Interrupt 半自动 | ❌ 无 | ❌ 手动 |

## 七、对 Agent Platform 的启示

1. **并发模型**：LangGraph 的 BSP 超步并行 + Send 扇出模式值得借鉴——同一阶段内独立任务可并行执行，完事后聚合，比当前简单的 asyncio.gather 更结构化。
2. **熔断机制**：五大竞品均未提供生产级熔断器，这正是我们的差异化机会。建议参考 LangGraph 的 Interrupt+checkpointer 模式，在 Worker v5.0 现有 600s 超时基础上增加：按 Agent 粒度的错误率阈值自动降级、LLM 调用失败自动切换备用模型（已有 fallback 链可扩展）。
3. **实时推送**：LangGraph 的 SSE `stream_mode` 和 ChatDev 2.0 的 WebSocket 日志面板是最成熟的参考。我们已有的 SSE 推送可增强为多模式（状态快照 + token 流 + 工具调用事件），类似 LangGraph 的分层流式设计。
4. **编排模式**：ChatDev 2.0 的可视化拖拽 + RL 优化编排路径是最前沿的方向。短期可对标 CrewAI Flows 的事件驱动 DAG 装饰器模式，中期引入可视化编排画布，长期探索 RL 自动优化编排拓扑。
